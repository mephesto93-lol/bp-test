## 1. Цель и область применения

### 1.1. Назначение системы

In-Plan — портал для визуального конструирования, управления и мониторинга бизнес-процессов (БП) и цепочек процессов организации. Система позволяет:

- проектировать процессы в визуальном редакторе (drag-and-drop, BPMN-подобная нотация);
- публиковать и запускать шаблоны процессов как экземпляры;
- назначать задачи пользователям с контролем SLA;
- автоматически запускать процессы по расписанию;
- отслеживать ход выполнения на диаграмме Ганта;
- выстраивать цепочки алгоритмов с фильтрами и конфигурацией.

### 1.2. Основные понятия

| Термин | Определение |
|--------|-------------|
| **Шаблон процесса** | Описание процесса — узлы, связи, настройки. Аналог "класса" в программировании |
| **Экземпляр** | Конкретный запуск шаблона — имеет свой прогресс, ответственного, сроки |
| **Узел** | Элемент процесса: задача, шлюз, начало, конец |
| **Связь** | Стрелка между узлами, определяет маршрут выполнения |
| **Алгоритм** | Автоматизированный шаг (скрипт, интеграция, SQL), не требующий ручного выполнения |
| **Цепочка процессов** | Тип процесса, состоящий только из алгоритмов, с глобальным тайм-аутом и логами |
| **SLA** | Время, отведённое на выполнение шага |
| **Критический путь** | Самый длинный по суммарному SLA маршрут от Старта до Конца |

## 2. Роли и доступ

### 2.1. Текущие роли (MVP)

В MVP системы предусмотрена единая роль пользователя. Разграничение прав планируется в последующих версиях.

### 2.2. Ролевая модель для алгоритмов (заложено в архитектуру)

Для **Цепочек процессов** предусмотрена следующая модель:

- Каждая цепочка имеет **сервисный аккаунт**, от имени которого выполняются алгоритмы.
- Пользователь управляет правом **запустить цепочку целиком** — это отдельное разрешение.
- Алгоритмы внутри цепочки выполняются от сервисного аккаунта, независимо от прав инициатора.
- При попытке запуска система **предварительно проверяет** наличие у пользователя права запуска цепочки и доступность всех алгоритмов (через сервисный аккаунт).
- Если у пользователя нет права запуска — система **блокирует старт** и показывает список недоступных шагов до начала выполнения.

**Заголовок секции:** «Шаблоны бизнес процессов»

**Кнопки действий:**

| Кнопка | Условие активности | Действие |
|--------|--------------------|----------|
| **+ (создать)** | Всегда | Открывает визуальный редактор в режиме создания |
| **📋 Копировать** | Выбран шаблон | Создаёт копию с суффиксом `_2`, `_3`… в статусе «Черновик» |

| **📦 Архивировать** | Выбран шаблон со статусом «Активен» | Открывает диалог подтверждения, после подтверждения переводит шаблон в статус «Архив», отключает автозапуск. Активные экземпляры продолжают выполняться, новые запустить нельзя. |

**Диалог подтверждения архивирования:**
- Текст: «Архивировать шаблон «[Название]»? Активные экземпляры продолжат работать. Новые запустить будет нельзя. Автозапуск будет отключён.»
- Кнопки: «Отмена» / «Подтвердить»

**Колонки таблицы:**

| Колонка | Тип | Описание |
|---------|-----|----------|
| ID | Число | Уникальный идентификатор шаблона |
| Имя процесса | Текст | Название + бейдж «🔗 Цепочка» если тип = Цепочка процессов |
| Владелец | Текст | ФИО владельца |
| Экземпляры | Число | Количество запущенных экземпляров |
| Дата обн. | Дата | Дата последнего обновления (ДД.ММ.ГГГГ) |
| Статус | Badge | Активен / Черновик / Архив |
| Автозапуск | Текст | Включен / Выключен |
| — | Иконка 🔍 | Открывает конструктор в режиме просмотра схемы |

**Поведение:**
- Клик на строку — выбирает шаблон, обновляет секцию экземпляров ниже.
- Выбранная строка подсвечивается фиолетовым.

## 5. Конструктор — визуальный редактор

### 5.1. Режимы редактора

| Режим | Когда открывается | Возможности |
|-------|-------------------|-------------|
| **create** | По нажатию «+» | Полное редактирование: узлы, связи, свойства, публикация |
| **edit-meta** | По 🔍 для Активного шаблона | Редактирование метаданных узлов (инструкции, исполнители); просмотр свойств **всех** типов узлов (условия XOR, параметры алгоритмов, режим AND); нельзя добавлять/удалять узлы и связи |
| **view** | По 🔍 для Архивного шаблона | Только чтение; просмотр свойств **всех** типов узлов: инструкции, исполнители, условия XOR, параметры алгоритмов, режим AND |

### 5.2. Верхняя панель (шапка)

- **← Назад** — закрыть редактор, вернуться на главный экран
- **Поле названия** — редактируемое в режиме `create`, read-only в остальных
- **Badge статуса** — Черновик (оранжевый) / Активен (зелёный) / Архив (серый)
- Подсказка горячих клавиш: `Del` — удалить узел / разгруппировать, `Esc` — отменить связь, `Shift+клик` — выбрать несколько узлов для группировки

### 5.3. Три панели редактора

```
┌──────────────┬──────────────────────────────┬──────────────────┐
│  Иерархия    │     Палитра + Холст           │    Свойства      │
│  (220px)     │     (flex:1)                  │    (280px)       │
└──────────────┴──────────────────────────────┴──────────────────┘
```

Каждую панель можно скрыть/показать кнопками ◀ ▶ внизу.

### 5.4. Левая панель: Иерархия

Отображает дерево узлов процесса. Если в процессе есть группы, они отображаются как папки, а их узлы — с дополнительным отступом:

```
📄 Название процесса        ← клик → показывает настройки процесса в правой панели
  📁 Блок согласования      ← клик → выделяет группу, открывает свойства группы
      🔷 Проверка условий
      🔷 Юридическое согласование
  🟢 Старт                  ← несгруппированные узлы отображаются на верхнем уровне
  🔶 Решение (XOR)
  🔴 Конец
```

- Клик на узел → выделяет его на холсте и открывает его свойства в правой панели.
- Клик на группу (📁) → выделяет группу на холсте и открывает свойства группы в правой панели.
- Выбранный узел / группа подсвечивается фиолетовым.

### 5.5. Центральная панель: Палитра + Холст

**Палитра** (левый край холста, вертикальная полоска 66px):

Сгруппирована по категориям:

| Категория | Элемент | Значок | Описание |
|-----------|---------|--------|----------|
| **События** | Старт | ▶ | Начало процесса |
| **События** | Конец | ■ | Завершение процесса |
| **Действия** | Алгоритм | `</>` | Автоматический шаг |
| **Действия** | Согласование | 👤 | Ручная задача для исполнителя |
| **Логика** | XOR | X | Условный шлюз — маршрутизация по условию |
| **Логика** | AND | + | Параллельный шлюз — разделение/слияние |

- Элементы перетаскиваются (drag-and-drop) на холст.
- В режимах `edit-meta` и `view` палитра скрыта.

**Холст:**

- Фоновая сетка 24×24px (серый цвет).
- Масштабирование: кнопки 🔍− / 🔍+ внизу или Ctrl+колесо мыши.
- Перемещение (pan): зажать пустое место и тянуть.
- Масштаб отображается в % в центре нижней панели.

**Взаимодействие с узлами:**

| Действие | Результат |
|----------|-----------|
| Клик на узел | Выделяет узел, показывает его свойства в правой панели |
| `Shift` + клик на узел | Добавляет / снимает узел в мульти-выделение (синяя рамка); повторный Shift+клик снимает |
| Зажать и тянуть узел | Перемещение узла по холсту |
| Навести на узел → появляется фиолетовая точка | Начало рисования связи |
| Тянуть от точки к другому узлу | Создаёт связь |
| Нажать `Delete` при выбранном узле | Удалить узел (кроме Старта) |
| Нажать `Delete` при выбранной группе | Разгруппировать (рамка исчезает, узлы остаются) |
| Нажать `Escape` при рисовании связи | Отменить рисование |
| Клик на пустое место холста | Снять выделение, показать настройки процесса |
| Клик на фоне группы (пунктирная рамка) | Выделить группу, показать свойства группы |
| Зажать и тянуть фон группы | Переместить все узлы группы одновременно |

**Отображение узлов:**

- **start / end** — круги (диаметр 48px)
- **task-approval / task-script** — прямоугольники (140×60px), с иконкой типа в левом верхнем углу и крупным названием по центру
- **gw-xor / gw-and** — ромбы (48×48px)

**Отображение связей (SVG):**

- Обычная связь — серая сплошная стрелка
- Связь «По умолчанию» (default) — оранжевая пунктирная стрелка; метка на схеме: **«↙ По умолчанию»**
- Рисуемая связь — фиолетовая пунктирная линия, следующая за курсором

### 5.6. Правая панель: Свойства узла

Показывает свойства выбранного узла. Если ни один узел не выбран — свойства процесса.

#### Состав панели в зависимости от режима и типа узла

| Тип узла | create | edit-meta | view |
|----------|--------|-----------|------|
| **Процесс** | Все настройки: название, описание, владелец, тип, тайм-аут, цикл | Описание, владелец, расписание (если есть) | Все поля read-only |
| **Старт** | Описание, тип триггера, расписание | Описание | Описание, тип триггера |
| **Конец** | Описание | Описание | Описание |
| **Согласование** | Инструкция, исполнитель, SLA, исходы, уведомления | Инструкция, исполнитель | Все поля read-only |
| **Алгоритм** | Алгоритм, версия, параметры, фильтры, поведение при ошибке | Алгоритм, версия, параметры, фильтры, поведение при ошибке (read-only) | Алгоритм, версия, параметры, фильтры, поведение при ошибке (read-only) |
| **XOR-шлюз** | Условия на каждую исходящую связь, выбор типа «Условный» / «По умолчанию» | Список исходящих связей с условиями (read-only) + предупреждение | Список исходящих связей с условиями (read-only) |
| **AND-шлюз** | Режим работы (Fork / Join / Не определён) с автоопределением | Режим работы (read-only) | Режим работы (read-only) |

**Для Алгоритма** панель включает (во всех режимах, кроме create — read-only):
- Алгоритм (название)
- Версия алгоритма
- Тайм-аут (мин.)
- ⚙️ Конфигурация: специфичные параметры выбранного алгоритма
- 🔍 Фильтры данных: ограничения набора данных
- При ошибке / тайм-ауте: «Остановить цепочку» / «Пропустить шаг и продолжить»

**Для XOR-шлюза** в режимах `edit-meta` и `view` панель показывает:
- Предупреждение: «Редактирование условий доступно в режиме «Редактировать»»
- Секцию «Исходящие связи» — для каждой связи:
  - Название целевого узла
  - Если тип = «По умолчанию»: оранжевая пометка «↙ По умолчанию — срабатывает если ни одно условие не выполнено»
  - Если тип = «Условный»: текст условия «Условие: «Шаг» оператор «Статус»»

### 5.7. Нижняя панель инструментов

- **◀** — скрыть/показать левую панель иерархии
- **▶** — скрыть/показать правую панель свойств
- **🔍−** — уменьшить масштаб
- **[N%]** — текущий масштаб
- **🔍+** — увеличить масштаб
- **⬡** — сгруппировать выбранные узлы (активна при мульти-выделении 2+ узлов)
- **⬢** — разгруппировать выбранную группу (активна при выбранной группе)

### 5.8. Группировка узлов

#### Назначение

Группировка позволяет объединить несколько узлов на холсте в именованный блок. Это улучшает читаемость сложных процессов, упрощает навигацию и позволяет перемещать связанные узлы одним действием.

#### Создание группы

1. Зажать `Shift` и кликнуть на два или более узла — они получают синюю рамку (мульти-выделение).
2. Нажать кнопку **⬡** в нижней панели инструментов.
3. На холсте появляется пунктирная синяя рамка вокруг выбранных узлов с названием **«Группа N»**.

#### Свойства группы (правая панель)

При клике на группу открывается панель свойств:

| Свойство | Описание |
|----------|----------|
| Название | Редактируемое текстовое поле; двойной клик на метке группы на холсте также открывает переименование |
| Узлы в группе | Список названий входящих узлов (read-only) |
| Кнопка «⬡ Разгруппировать» | Удаляет группу; узлы остаются на холсте |

#### Перемещение группы

- Клик и перетаскивание **фона** пунктирной рамки перемещает все узлы группы одновременно, сохраняя их взаимное расположение.
- Перемещение отдельного узла внутри группы не изменяет состав группы — рамка автоматически пересчитывает своё положение и размер.

#### Разгруппирование

Группу можно расформировать тремя способами:
- Кнопка **⬢** в нижней панели (при выбранной группе).
- Кнопка **«⬡ Разгруппировать»** в правой панели свойств.
- Клавиша `Delete` при выбранной группе.

Узлы при этом остаются на холсте с сохранёнными связями.

#### Ограничения

- Группировка доступна только в режиме `create`.
- В режимах `edit-meta` и `view` группы не отображаются и недоступны для редактирования.
- Один узел может входить только в одну группу (вложенные группы не поддерживаются).
- Минимальный состав группы — 2 узла.

#### Отображение в иерархии

Группы отображаются как папки 📁 в левой панели «Иерархия». Узлы-члены группы показаны с дополнительным отступом под своей группой. Несгруппированные узлы располагаются ниже групп в порядке топологической сортировки.

## 6. Типы узлов и их свойства

### 6.1. Старт (start)

**Форма:** Зелёный круг
**Обязателен:** Да, ровно один на процесс

| Свойство | Тип | Обязательное | Описание |
|----------|-----|:---:|----------|
| Название | Текст | ✓ | Отображается на холсте |
| Описание | Textarea | — | Условие или контекст запуска |
| Тип триггера | Select | ✓ | «Ручной запуск» / «По расписанию» |
| Расписание | Текст | Если «По расписанию» | Описание расписания, напр. «09:00 каждый ПН» |

> Расписание детально настраивается в разделе «Цикл повторений» настроек процесса.

### 6.2. Конец (end)

**Форма:** Красный круг с двойной границей
**Обязателен:** Да, минимум один

| Свойство | Тип | Обязательное | Описание |
|----------|-----|:---:|----------|
| Название | Текст | ✓ | Отображается на холсте |
| Описание | Textarea | — | Условие завершения |

### 6.3. Согласование (task-approval)

**Форма:** Голубой прямоугольник, иконка 👤
**Назначение:** Ручная задача, требующая действия от исполнителя

| Свойство | Тип | Обязательное | Описание |
|----------|-----|:---:|----------|
| Название | Текст | ✓ | Отображается на холсте |
| Инструкция | Textarea | ✓ | Текст поручения для исполнителя |
| Тип исполнителя | Select | ✓ | «Конкретный пользователь» / «Роль» / «Группа» |
| Исполнитель | Текст | ✓ | Имя, название роли или группы |
| SLA — значение | Число (≥1) | ✓ | Число единиц времени |
| SLA — единица | Select | ✓ | «минут» / «часов» / «дней» |
| Исходы | Список строк | ✓ (мин. 1) | Кнопки действия для пользователя; каждый исход формирует ветку маршрута |
| Создавать карточку в Task Tracker | Checkbox | — | При включении создаётся задача в трекере при активации узла |

**Управление исходами:**
- Кнопка «+ Добавить исход» — добавляет пустое поле
- Кнопка удаления (×) рядом с каждым исходом
- Минимум один исход обязателен

### 6.4. Алгоритм (task-script)

**Форма:** Синий прямоугольник, иконка ⚙
**Назначение:** Автоматический шаг — скрипт, интеграция, вычисление

| Свойство | Тип | Обязательное | Описание |
|----------|-----|:---:|----------|
| Название | Текст | ✓ | Отображается на холсте |
| Алгоритм | Select | ✓ | Выбор из реестра алгоритмов |
| Версия алгоритма | Select | Если алгоритм выбран | Зависит от выбранного алгоритма |
| Конфигурация | Набор полей | Зависит от алгоритма | Параметры, специфичные для каждого алгоритма |
| Фильтры данных | Набор полей | — | Ограничивают набор данных для алгоритма |
| Поведение при ошибке/тайм-ауте | Radio | ✓ | «Остановить цепочку» / «Пропустить шаг и продолжить» |

> Тайм-аут задаётся **глобально** в настройках процесса, а не на уровне конкретного шага.

### 6.5. Шлюз XOR (gw-xor)

**Форма:** Оранжевый ромб, буква X
**Назначение:** Условная маршрутизация — один из нескольких путей

| Свойство | Описание |
|----------|----------|
| Название | Отображается на холсте |
| Условия | Настраиваются на **исходящих связях**, не на самом узле |

**Требования:**
- Минимум одна исходящая связь типа «По умолчанию» (default)
- Остальные связи — условные, с явным условием

### 6.6. Шлюз AND (gw-and)

**Форма:** Оранжевый ромб, знак +
**Назначение:** Параллельное разделение или слияние потоков

**Режим определяется автоматически по топологии:**

| Режим | Условие | Поведение |
|-------|---------|-----------|
| **Fork (разделение)** | 1 входящая, 2+ исходящих | Все исходящие ветки запускаются параллельно |
| **Join (слияние)** | 2+ входящих, 1 исходящая | Ожидает завершения **всех** входящих веток |
| **Не определён** | Нет связей | Ошибка валидации |

**Требование:** Каждый AND-Fork должен иметь соответствующий AND-Join.

## 7. Связи и маршрутизация

### 7.1. Типы связей

| Тип | Внешний вид | Метка на схеме | Описание |
|-----|-------------|----------------|----------|
| **Обычная (conditional)** | Серая сплошная стрелка | — или значение conditionAttr | Стандартный переход |
| **По умолчанию (default)** | Оранжевая пунктирная стрелка | «↙ По умолчанию» | Используется в XOR, если ни одно условие не выполнено |

### 7.2. Свойства связи

| Свойство | Тип | Условие | Описание |
|----------|-----|---------|----------|
| Тип потока | Radio | Из XOR | «Условный» / «По умолчанию» |
| Шаг-источник условия | Select | Тип = Условный | Задача, результат которой проверяется |
| Оператор | Select | Тип = Условный | «= равно» / «≠ не равно» |
| Статус для проверки | Select | Тип = Условный | Зависит от типа выбранного шага (см. ниже) |

**Доступные значения «Статус для проверки» зависят от типа шага-источника:**

| Тип шага-источника | Доступные статусы | Подсказка в интерфейсе |
|--------------------|-------------------|------------------------|
| **Согласование** (task-approval) | Исходы этой задачи (настраиваются в свойствах узла) | «Исходы задачи "Название"» |
| **Алгоритм** (task-script) | Просрочена / Завершена / Запланирована / В работе / Отменена | «Системные статусы алгоритма» |
| Шаг не выбран | Просрочена / Завершена / Запланирована / В работе / Отменена | — |

> При смене шага-источника ранее выбранный статус сбрасывается автоматически.

**Пример условия (Согласование):** «Если шаг "Юридическое согласование" = Согласовано → идти на "Регистрацию договора"»

**Пример условия (Алгоритм):** «Если шаг "Расчёт прогноза" = Завершена → идти на "Экспорт в ERP"»

### 7.2.1. Механизм маршрутизации токена в runtime

Когда исполнитель нажимает кнопку исхода в задаче-Согласовании (например, «Согласовано»), движок процесса выполняет следующее:

1. Фиксирует результат узла: `{ nodeId, outcome: "Согласовано" }`.
2. Перебирает все исходящие связи этого узла.
3. Для каждой условной связи проверяет: `conditionAttr == outcome` (строгое совпадение строк).
4. Активирует **первую** связь, условие которой выполнено, и передаёт токен в целевой узел.
5. Если ни одна условная связь не совпала — токен идёт по связи типа **«По умолчанию»** (если она есть).

**Поведение при ошибке маршрутизации:**

| Ситуация | Поведение системы |
|----------|------------------|
| Найдена ровно одна подходящая связь | Токен переходит по ней |
| Найдено несколько подходящих связей | Токен идёт по первой (порядок — по времени создания связи) |
| Подходящей связи нет, есть «По умолчанию» | Токен идёт по «По умолчанию» |
| Подходящей связи нет, «По умолчанию» нет | Экземпляр процесса переходит в статус **«Ошибка»**, владелец получает уведомление |

> **Важно для конструктора:** строка исхода в свойствах узла и значение `conditionAttr` на связи должны совпадать **точно**. Интерфейс подставляет значение автоматически при выборе «Статус для проверки» из списка исходов узла — ручной ввод не допускается.

### 7.3. Создание и удаление связей

- Связь рисуется от **фиолетовой точки** на исходном узле до целевого узла
- Связь удаляется через свойства исходного узла
- `Escape` — отменяет рисование текущей связи

## 9. Валидация и публикация

### 9.1. Валидация схемы

Проверяется при каждой попытке публикации. При наличии ошибок публикация недоступна.

| Правило | Сообщение об ошибке |
|---------|---------------------|
| Нет узла «Старт» | «Добавьте узел "Старт"» |
| Нет узла «Конец» | «Процесс должен иметь узел "Конец"» |
| Узел без исходящей связи (кроме Конца) | «"[Название]": нет исходящей связи» |
| Узел без входящей связи (кроме Старта) | «"[Название]": нет входящей связи» |
| AND-Fork без AND-Join | «Параллельные ветви AND нужно свести — добавьте AND-шлюз слияния» |
| XOR без пути «По умолчанию» | «Шлюз "[Название]": укажите один путь "По умолчанию"» |

### 9.2. Валидация цикла (при автозапуске)

При публикации процесса с включённым циклом дополнительно проверяется:

```
Сумма SLA всех шагов критического пути ≥ Частота запуска цикла
```

Если условие выполняется — показывается **специальное модальное окно**:

- Заголовок: «⚠ Конфликт SLA и расписания»
- Сообщение: «Суммарный SLA процесса превышает частоту запуска цикла»
- Список задач критического пути с их SLA
- Рекомендации: «Уменьшите SLA задач, увеличьте частоту цикла или отключите автозапуск»
- Кнопки: «Назад к редактированию» / «Всё равно опубликовать»

**Расчёт критического пути:**
- Алгоритм: топологический обход от Старта до Конца, максимальная сумма SLA
- Шаги-алгоритмы (task-script) учитываются по глобальному тайм-ауту процесса

### 9.3. Валидация полей

| Поле | Правило | Сообщение |
|------|---------|-----------|
| Имя процесса | Уникальность | «⚠ Название уже используется. Измените имя.» |
| SLA (task-approval) | > 0 | Поле подсвечивается красным |
| Исходы (task-approval) | Минимум 1 | Подсказка: «Добавьте хотя бы один исход» |
| Алгоритм (task-script) | Выбран | Звёздочка * на метке |
| Дата начала цикла | Не в прошлом | Поле игнорирует прошлые даты |

## 10. Раздел: Мои задачи

### 10.1. Заголовок и KPI

**Заголовок:** «Мои задачи (N)» — N = число активных задач

**KPI-полоса:**

| Чип | Цвет | Условие отображения | Содержание |
|-----|------|---------------------|------------|
| 🔴 Просрочено | Красный | N > 0 | Число просроченных задач |
| 🔵 В работе | Синий | N > 0 | Число активных задач |
| ✓ Завершено | Зелёный | Всегда | Число завершённых задач |

### 10.2. Вкладка «Активные»

**Карточка задачи:**

```
● Согласовать договор с ООО «Альфа»           12.06 14:30 →
  Процесс: Согласование договора · Экземпляр E-101
```

| Элемент | Описание |
|---------|----------|
| Точка (●) | Серая — в сроке; Красная — просрочена |
| Название | Жирный текст |
| Подзаголовок | Процесс + ID экземпляра |
| Дедлайн | Дата и время; красный если просрочена |
| Стрелка (›) | Индикатор кликабельности |

**Поведение списка:**
- До 7 задач — показываются все
- Более 7 — показываются первые 7, кнопка «Показать все (N)»
- Нет задач — пустое состояние: «✓ На сегодня задач нет. Отличная работа!»

### 10.3. Вкладка «Завершённые»

Таблица с колонками:

| Колонка | Описание |
|---------|----------|
| Название задачи | Текст |
| Процесс | «Процесс — ID экземпляра» |
| Дата выполнения | ДД.ММ.ГГГГ ЧЧ:ММ |
| Результат | Badge с исходом задачи |

**Цвета badge результата:**
- Зелёный — Согласовано, Утверждено, Сделано, Выполнено и аналогичные
- Красный — Отклонено

Отображаются первые 30 записей.

### 10.4. Task Drawer (боковое окно задачи)

Открывается справа при клике на карточку.

**Секции:**

| Секция | Содержание |
|--------|------------|
| Заголовок | Тег «[Процесс] · [Экземпляр]», название задачи, кнопка ✕ |
| Метаинформация | Процесс, запустил, поступила (дата+время), дедлайн (красный + «⚠ Просрочена» если нужно) |
| Инструкция | Текст поручения для исполнителя |
| Действия | Кнопки исходов (основная — фиолетовая, остальные — белые с обводкой) |

**Логика кнопок исходов:**
- Первый исход — основная кнопка (primary)
- Остальные исходы — вторичные кнопки

**При нажатии на исход:**

| Цвет чипа | Значение |
|-----------|----------|
| Зелёный (✓) | Шаг выполнен |
| Фиолетовый (●) | Шаг в работе |
| Серый | Будущий шаг |

## 13. Модели данных

### 13.1. Шаблон процесса

```
Template {
  id:             number          // Уникальный ID
  name:           string          // Название шаблона
  owner:          string          // Владелец (ФИО)
  count:          number          // Кол-во экземпляров
  date:           string          // Дата обновления (ДД.ММ.ГГГГ)
  status:         'Активен' | 'Черновик' | 'Архив'
  auto:           'Включен' | 'Выключен'
  type:           'БП' | 'Цепочка'
  currentVersion: string?         // Только для type='Цепочка'
  versions:       Version[]?      // Только для type='Цепочка'
}
```

### 13.2. Версия цепочки

```
Version {
  ver:    string   // Код версии (v1.0, v2.1…)
  date:   string   // Дата фиксации (ДД.ММ.ГГ)
  note:   string   // Описание изменений
  status: 'Активна' | 'Архив'
}
```

### 13.3. Экземпляр процесса

```
Instance {
  id:       string   // E-101, Z-001, O-201…
  start:    string   // Дата старта (ДД.ММ.ГГ)
  user:     string   // ФИО ответственного
  status:   'Активен' | 'Просрочен' | 'Завершен'
  progress: number   // 0–100
  deadline: string   // Дедлайн (ДД.ММ.ГГ)
  overdue:  number   // Дней просрочки (0 если нет)
  end:      string   // Дата завершения или '—'
}
```

### 13.3.1. Расчёт прогресса экземпляра

```
progress = завершённые_шаги / всего_шагов × 100
```

**Всего шагов** — количество узлов типа `task-approval` и `task-script` в шаблоне. Узлы `start`, `end`, `gw-xor`, `gw-and` не учитываются.

**Завершённые шаги** — записи в `TaskLog` данного экземпляра, где `completedAt != null`.

> При маршрутизации через XOR-шлюз учитываются только фактически пройденные шаги, а не все ветки схемы. Знаменатель (всего шагов) при этом остаётся фиксированным — равным полному числу `task-approval` и `task-script` в шаблоне.

### 13.4. Запись журнала задач

```
TaskLog {
  step:        string             // Название шага

  startedAt:   string             // 'ДД.ММ ЧЧ:ММ'
  completedAt: string | null      // null если не завершён
  onTime:      boolean | null     // null если не завершён
  outcome:     string | null      // Результат или null
  current:     boolean?           // true для текущего шага
}
```

### 13.5. Узел схемы

```
Node {
  id:    number
  type:  'start' | 'end' | 'task-approval' | 'task-script' | 'gw-xor' | 'gw-and'
  label: string
  x:     number   // Позиция на холсте
  y:     number
  props: {
    // start
    description?: string
    triggerType?: 'Ручной запуск' | 'По расписанию'
    schedule?:    string

    // end
    description?: string

    // task-approval
    instruction?:   string
    assigneeType?:  'Конкретный пользователь' | 'Роль' | 'Группа'
    assignee?:      string

    slaUnit?:       'минут' | 'часов' | 'дней'

    taskTracker?:   boolean

    algorithm?:  string
    version?:    string
    params?:     Record<string, any>
    filters?:    Record<string, any>

  }
}
```

```
}
```

```
Task {
  id:          string   // T-001, T-002…
```

## 16. Ролевая модель доступа к алгоритмам

Данный раздел описывает архитектурный принцип, который должен быть реализован в production-версии системы.

### 16.1. Принцип двухуровневого контроля доступа

**Уровень 1: Право запустить цепочку**
- Проверяется перед стартом цепочки
- Назначается пользователю или роли
- При отсутствии права — запуск заблокирован, пользователю показывается список недоступных шагов

**Уровень 2: Права выполнения алгоритмов**
- Алгоритмы выполняются от имени **сервисного аккаунта** цепочки
- Сервисный аккаунт задаётся при настройке цепочки администратором
- Пользователь, запустивший цепочку, не может иметь больше прав, чем сервисный аккаунт

### 16.2. Проверка перед запуском

При нажатии «▶ Запустить» в Менеджере процессов или при автозапуске:

1. Проверить: есть ли у пользователя право **запускать** данную цепочку
2. Проверить: есть ли у сервисного аккаунта цепочки доступ ко **всем алгоритмам** в схеме
3. Если проверка 1 не пройдена → показать ошибку с пояснением
4. Если проверка 2 не пройдена → показать список алгоритмов без доступа
5. Если всё ок → запустить экземпляр

## 17. Система уведомлений

### 17.1. Назначение

Уведомления информируют пользователей о событиях в процессах: назначении задач, приближении дедлайнов, просрочках, завершении экземпляров и ошибках в цепочках.

### 17.2. Каналы доставки

| Канал | Описание |
|-------|----------|
| **In-app** | Колокольчик 🔔 в шапке портала. Счётчик непрочитанных. Панель со списком уведомлений |
| **Email** | Отправка письма на почту пользователя |

Пользователь самостоятельно выбирает каналы в личных настройках уведомлений.

### 17.3. Центр уведомлений (In-app)

**Элементы:**
- Иконка 🔔 в заголовке приложения
- Красный бейдж с числом непрочитанных (скрывается при 0)
- Клик → выпадающая панель со списком уведомлений

**Панель уведомлений:**
- Список уведомлений в обратном хронологическом порядке
- Непрочитанные — фиолетовый фон + точка + жирный текст
- Прочитанные — белый фон, обычный текст
- Клик на уведомление → помечает как прочитанное
- Кнопка «Прочитать все»
- Кнопка «⚙ Настройки уведомлений» → открывает модальное окно настроек

### 17.4. Типы событий и уведомлений

| Тип | Иконка | Кому | Описание |
|-----|--------|------|----------|
| **Назначение задачи** | 👤 | Исполнителю | При переходе узла task-approval в активный статус |
| **Просрочка задачи** | ⚠️ | Исполнителю + владельцу процесса | При превышении SLA задачи |
| **Напоминание о дедлайне** | ⏰ | Исполнителю | За N часов до дедлайна задачи |
| **Завершение экземпляра** | ✅ | Владельцу процесса / инициатору | При завершении всего экземпляра |
| **Ошибка алгоритма** | 🔴 | Владельцу процесса | При падении шага task-script с ошибкой |
| **Запуск цепочки** | 🔗 | Инициатору / владельцу | При успешном старте цепочки процессов |

### 17.5. Настройки уведомлений — три уровня

#### Уровень 1: Личные настройки пользователя

Доступны через кнопку «⚙ Настройки уведомлений» в панели уведомлений.

| Настройка | Тип | Описание |
|-----------|-----|----------|
| In-app уведомления | Checkbox | Включить/выключить канал In-app |
| Email уведомления | Checkbox | Включить/выключить канал Email |
| Новая задача назначена мне | Checkbox | — |
| Задача просрочена | Checkbox | — |
| Напоминание о приближающемся дедлайне | Checkbox | — |
| За N часов до дедлайна | Число (1–72) | Значение по умолчанию: 24 часа |
| Экземпляр процесса завершён | Checkbox | — |
| Ошибка в цепочке процессов | Checkbox | — |

> Персональные настройки имеют **наименьший приоритет** — могут быть переопределены настройками на уровне процесса или узла.

#### Уровень 2: Настройки процесса

Раздел «🔔 Уведомления процесса» в настройках процесса (конструктор → правая панель → настройки процесса).

| Настройка | Тип | Варианты |
|-----------|-----|----------|
| Канал уведомлений | Select | «In-app + Email» / «Только In-app» / «Только Email» / «Отключены» |
| Уведомлять владельца при завершении экземпляра | Checkbox | — |
| Уведомлять владельца при ошибке или падении алгоритма | Checkbox | — |
| Уведомлять владельца при просрочке любой задачи | Checkbox | — |

> Настройки процесса **переопределяют** канал для всех событий, связанных с данным процессом.

#### Уровень 3: Настройки узла (task-approval)

Раздел «🔔 Уведомления» в свойствах узла Согласование.

| Настройка | Тип | Описание |
|-----------|-----|----------|
| Уведомить исполнителя при назначении | Checkbox | По умолчанию включено |
| Напоминание за N часов до дедлайна | Checkbox | Активирует поле ввода часов |
| Количество часов | Число (1–72) | По умолчанию 24 |

> Настройка узла **переопределяет** персональное значение напоминания для данного конкретного шага.

### 17.6. Приоритет настроек

```
Настройки узла > Настройки процесса > Персональные настройки пользователя
```

Если на уровне процесса канал = «Отключены» — уведомления не отправляются независимо от настроек узла и пользователя.

### 17.7. Модель данных уведомления

```
Notification {
  id:      number
  read:    boolean
  icon:    string          // Эмодзи-иконка типа события
  title:   string          // Заголовок уведомления
  sub:     string          // Подзаголовок (процесс, экземпляр)
  time:    string          // Относительное время («5 мин. назад»)
  type:    'task' | 'overdue' | 'done' | 'reminder' | 'chain' | 'error'
}
```

## 18. Нефункциональные требования

### 17.1. UX и интерфейс

- Все действия пользователя должны иметь мгновенный отклик (< 100 мс для UI)
- Toast-уведомления появляются внизу справа и исчезают через 3 секунды
- Модальные окна закрываются кликом по тёмному фону (backdrop)
- Drawer-панели анимируются (slide-in справа, 200 мс)
- Все таблицы поддерживают выделение строки (hover + active состояния)

### 17.2. Конструктор

- Холст поддерживает zoom от 20% до 200%
- Drag-and-drop узлов работает на мобильных устройствах (touch events)
- Связи перерисовываются в реальном времени при перемещении узлов
- Отмена текущей связи (Escape) должна немедленно убирать временную линию

### 17.3. Данные

- Имена процессов уникальны в рамках всего реестра шаблонов
- ID экземпляров уникальны глобально
- Все даты хранятся в формате ДД.ММ.ГГ, отображаются в локали пользователя
- Прогресс экземпляра — целое число 0–100

### 17.4. Совместимость

- Поддерживаемые браузеры: Chrome 100+, Firefox 100+, Edge 100+
- Минимальное разрешение экрана: 1280×720
- Конструктор оптимизирован для разрешений от 1440px по ширине
